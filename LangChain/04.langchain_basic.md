# 📅 2026-03-26 실무 밀착형 학습 기록

## 📋 학습 개요
오늘은 로컬 환경에서 LLM을 구동하는 방법과 RAG(검색 증강 생성) 시스템의 핵심 파이프라인인 **문서 처리(Document Processing)** 및 **벡터 데이터베이스(Vector DB)** 활용법을 학습했습니다.

---

## 🚀 1. 로컬 LLM 및 Ollama 활용
API 비용 걱정 없이 내 컴퓨터에서 직접 모델을 돌리는 방법을 익혔습니다.

* **Ollama**: 로컬에서 오픈소스 모델(Llama, Qwen 등)을 쉽게 실행하게 해주는 런타임입니다.
* **양자화(Quantization)**: 모델의 가중치를 압축하여 일반 PC에서도 빠르게 돌아가게 만드는 기술입니다.
* **실무 포인트**: 보안이 중요한 프로젝트나 간단한 테스트 단계에서는 로컬 모델을 쓰고, 고도의 추론이 필요한 서비스 단계에서는 GPT-4o 같은 API 모델로 교체하는 유연한 설계가 필요합니다.

---

## 🏗️ 2. RAG 파이프라인: 문서 로드와 분할
데이터를 인공지능이 읽기 좋은 형태로 가공하는 단계입니다.

### 📄 Document Loader & Text Splitter
* **PyPDFLoader**: PDF 문서를 페이지 단위로 불러옵니다.
* **RecursiveCharacterTextSplitter**: 문맥을 최대한 보존하며 데이터를 자르는 도구입니다.
    * `chunk_size`: 잘라낸 조각의 크기입니다.
    * `chunk_overlap`: 조각 간의 중복 부분으로, **문맥의 끊김을 방지**하는 안전장치입니다. (보통 size의 10~20% 권장)

---

## 🧠 3. 임베딩(Embedding)과 유사도 검색
텍스트의 '의미'를 숫자로 바꿔서 컴퓨터가 이해하게 만드는 과정입니다.

* **Embedding**: 단어나 문장을 다차원 공간의 좌표(Vector)로 변환합니다. 의미가 비슷하면 좌표상 거리도 가깝습니다.
* **코사인 유사도(Cosine Similarity)**: 두 벡터가 가리키는 방향이 얼마나 비슷한지 측정하여 유사성을 판단합니다.
* **핵심**: 임베딩 모델을 바꾸면 좌표 체계가 완전히 달라지므로, 반드시 **벡터 DB를 새로 구축**해야 합니다.

---

## 💾 4. 벡터 데이터베이스: Chroma DB
생성된 벡터들을 효율적으로 저장하고 '가장 비슷한 것'을 빠르게 찾아주는 저장소입니다.

* **ANN(Approximate Nearest Neighbor)**: 모든 데이터를 전수 조사하지 않고, 인덱스를 통해 빠르게 근삿값을 찾아내는 알고리즘입니다.
* **메타데이터(Metadata)**: 벡터에 붙이는 '이름표'입니다. 출처나 페이지 번호를 저장하여 검색 결과의 신뢰도를 높이고 필터링에 활용합니다.

## 🛠️ 실무 예시 코드 (Chroma DB 구축 및 검색)

실무 표준에 따라 구조화된 벡터 DB 구축 예시입니다.

```python
from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings
from langchain_community.document_loaders import PyPDFLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter

# 1. 문서 로드 및 청킹 전략 수립
file_path = "data/sample_document.pdf"
loader = PyPDFLoader(file_path)
raw_documents = loader.load()

# 기술 문서 특성에 맞춰 chunk_size 500, overlap 50(10%) 설정
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500, 
    chunk_overlap=50
)
document_chunks = text_splitter.split_documents(raw_documents)

# 2. 임베딩 모델 설정 및 벡터 DB 저장
# 가성비가 좋은 text-embedding-3-small 모델 사용
embedding_model = OpenAIEmbeddings(model="text-embedding-3-small")

vector_database = Chroma.from_documents(
    documents=document_chunks,
    embedding=embedding_model,
    collection_name="tech_support_v1",
    persist_directory="./chroma_storage"
)

# 3. 데이터 검색 (Retriever 인터페이스 활용)
search_query = "AI 모델의 성능 비교 지표는?"
retrieved_docs = vector_database.similarity_search(search_query, k=3)

for i, doc in enumerate(retrieved_docs):
    print(f"[{i+1}] 검색된 내용: {doc.page_content[:100]}...")
```

## ⚠️ 트러블슈팅 (Troubleshooting)

### 1. 임베딩 모델 불일치 이슈
* **문제**: 기존에 저장된 데이터와 현재 검색에 사용하는 임베딩 모델이 다를 경우 검색 결과가 전혀 엉망으로 나옴.
* **해결**: 모델명을 환경 변수나 설정 파일에 명시하고, 모델 변경 시 반드시 `persist_directory`를 비우고 다시 임베딩을 진행함.

### 2. 청크 경계에서의 문맥 단절
* **문제**: `chunk_overlap`이 0일 때, 중요한 단어가 반으로 잘려 검색 효율이 급격히 떨어짐.
* **해결**: 최소 10% 이상의 오버랩을 설정하여 앞뒤 문맥이 겹치도록 조정함.
