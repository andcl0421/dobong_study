## 🏗️ 1. Plan-and-Execute: "먼저 계획하고 실행하라"

ReAct 방식이 한 단계씩 앞만 보고 간다면, **Plan-and-Execute**는 지도 전체를 먼저 보고 움직입니다.

### 💡 핵심 설계 포인트
* **Planner (기획자):** 사용자의 요청을 분석해 최대 5단계 내외의 `list[str]` 계획을 생성합니다.
* **Executor (실행자):** 계획의 현재 단계를 수행합니다. 이때 이전 단계의 결과물인 `past_steps`를 참고하여 맥락을 유지합니다.
* **Replanner (조정자):** 실행 결과를 보고 "계획대로 계속할지", "계획을 수정할지", 아니면 "충분하니 답변을 줄지" 결정합니다.

> **멘토의 한마디:** `past_steps`에는 반드시 `operator.add` 리듀서를 사용해 데이터를 누적하세요. 그래야 뒤에 실행되는 노드가 앞선 결과를 알고 똑똑하게 동작합니다.

## 🤝 2. Multi-Agent 협업 패턴 (Supervisor vs Swarm)

에이전트가 너무 많은 도구(Tool)를 가지면 성능이 떨어집니다. 이를 해결하기 위해 전문 분야별로 에이전트를 나눕니다.

### 🏢 Supervisor (중앙 관리자형)
관리자 LLM이 팀원(Worker)들에게 업무를 배분합니다.
* **구조:** `사용자 → Supervisor → Worker → 다시 Supervisor`
* **장점:** 전체 흐름 제어가 한 곳에서 일어나므로 디버깅과 관리가 쉽습니다.
* **적합:** 정해진 절차나 검수 과정이 엄격한 보고서 작성 프로젝트 등.

### 🐝 Swarm (자율 협업형)
관리자 없이 에이전트끼리 직접 바톤을 넘깁니다.
* **구조:** `Agent A → (handoff) → Agent B → (handoff) → Agent C`
* **장점:** `Command(goto="다음노드")`를 사용해 동적으로 흐름을 넘기므로 유연하고 자연스럽습니다.
* **적합:** 여행 예약, 쇼핑 상담 등 상황에 따라 상담원이 바뀌어야 하는 서비스.

## 🛠️ 실무형 트러블슈팅 (Troubleshooting)

### 이슈 1: 컨텍스트 윈도우 폭발 (Context Overflow)
* **현상:** 멀티스텝을 거치면서 `past_steps`나 메시지가 너무 길어져 비용이 급증하고 모델이 느려짐.
* **해결:** 1.  **Truncation:** 최근 3~5개의 결과만 남기고 나머지는 요약본으로 대체합니다.
    2.  **Summary:** 각 단계가 끝날 때마다 LLM에게 "핵심만 요약해"라고 시킨 뒤 저장합니다.

### 이슈 2: 무한 루프 (Infinite Loop)
* **현상:** Swarm 패턴에서 에이전트끼리 서로 일을 미루며 핑퐁 게임을 함.
* **해결:** `recursion_limit`을 20 정도로 설정하여 강제 종료 장치를 마련하고, 프롬프트에 "이미 시도한 방법이 실패했다면 중단하라"는 지침을 명시합니다.

    final_response: str

# 2. 에러 대응을 위한 안전한 실행 노드 (Wrapper)
def safe_execute_node(state: PlanExecuteState):
    current_step = state["current_plan"][0]
    try:
        # 실제 에이전트 실행 로직 (예시)
        result = agent_executor.invoke({"input": current_step})
        content = result["output"]
    except Exception as error:
        # 에러가 나도 그래프가 죽지 않게 텍스트로 변환하여 저장
        content = f"시스템 오류 발생: {str(error)}"
    
    return {"execution_history": [(current_step, content)]}
