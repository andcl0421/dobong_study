# 🚀 실무 밀착형 멘토링: Agentic RAG 기반 챗봇 서비스 구축

비전공자 신입 개발자분들이 실무에서 가장 많이 접하게 될 **'스스로 생각하고 교정하는 AI'**인 Agentic RAG와 이를 서비스화하는 과정을 정리했습니다. 단순히 코드를 짜는 것을 넘어, 왜 이 기술이 필요한지 비즈니스 로직의 핵심을 파악하는 것이 중요합니다.

---

## 1. Agentic RAG vs RAG Agent
두 개념의 차이는 **'자율성의 깊이'**에 있습니다.

* **RAG Agent:** AI가 필요할 때 검색 도구(Retriever)를 꺼내 쓰는 수준입니다. 하지만 검색 결과가 나쁘더라도 그대로 믿고 답변하는 '단순함'이 한계입니다.
* **Agentic RAG:** 검색 결과가 질문과 관련이 있는지 **직접 채점(Grade)**하고, 내용이 부족하면 **질문을 수정(Rewrite)**해서 다시 찾아옵니다. 답변을 만든 후에도 **검증(Evaluate)** 과정을 거치는 자율 루프가 핵심입니다.

---

## 2. 설계 핵심 포인트: LangGraph 워크플로우
에이전트의 사고 과정을 설계할 때는 아래와 같은 **상태(State) 관리**가 가장 중요합니다. 특히 재시도 횟수를 제한하여 무한 루프에 빠지지 않게 설계해야 합니다.

````python
class AgenticRAGState(TypedDict):
    question: str          # 사용자 질문 (재작성 가능)
    documents: List[str]   # 검색된 문서들
    generation: str        # 생성된 답변
    relevance: bool        # 문서 관련성 여부
    is_sufficient: bool    # 답변 품질 만족 여부
    retry_count: Annotated[int, operator.add] # 재시도 횟수 (누적 합산)

    3. MCP (Model Context Protocol): AI의 USB-C
실무에서 Gmail, Slack, Notion 등을 일일이 API 연동하려면 코드가 수천 줄이 됩니다. MCP는 이를 표준화하여 서버 하나만 연결하면 모든 에이전트 도구를 즉시 사용하게 해주는 혁신적인 프로토콜입니다.

장점: Cursor, Claude Code, LangGraph 어디서든 동일한 도구를 재사용할 수 있습니다.

실무 팁: 이미 잘 만들어진 MCP 서버(Gmail, Google Search 등)는 가져다 쓰고, 우리 회사만의 고유 로직은 @tool로 직접 만드는 것이 효율적입니다.

4. 서비스 아키텍처: FastAPI & React
에이전트를 실제 웹 서비스로 만들 때는 **사용자 경험(UX)**을 위해 스트리밍 기술이 필수입니다.

SSE (Server-Sent Events) 스트리밍: 사용자는 AI가 답변을 다 쓸 때까지 기다려주지 않습니다. astream(stream_mode="messages")를 사용하여 토큰이 생성될 때마다 한 글자씩 화면에 뿌려주는 SSE 방식을 채택합니다.

💾 대화 내역 저장 (PostgreSQL & Checkpointer): 실제 주고받은 메시지 덩어리와 에이전트의 상태를 DB에 자동 저장합니다. thread_id만 알면 대화 맥락을 완벽히 복구할 수 있습니다.

🔍 트러블슈팅 (Troubleshooting)
이슈: 스트리밍이 작동하지 않고 답변이 한꺼번에 나옵니다.

원인: LangGraph 내부 노드 함수가 def와 llm.invoke()로 작성된 경우입니다. 동기 방식은 응답을 한 번에 다 받을 때까지 기다리기 때문에 중간 토큰을 뱉어낼 수 없습니다.

해결: 모든 노드를 async def로 선언하고, LLM 호출 시 반드시 await llm.ainvoke()를 사용해야 합니다. 비동기 환경을 맞춰야만 실시간 타이핑 효과가 구현됩니다.
